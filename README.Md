# Code Safety Checker

**Static analysis tool for detecting security vulnerabilities in AI-generated and human-written code.**

Live app: [https://starborn.github.io/codesafetycheckr2026/](https://starborn.github.io/codesafetycheckr2026/)

Authors: Paola Di Maio ([W3C AIKR CG](https://www.w3.org/community/aikr/) Chair) & Claude (Anthropic)
Affiliation: [Epistemic Systems Lab](https://ronin-institute.org), Ronin Institute / W3C AI Knowledge Representation CG
Reference: [w3c/webai#23](https://github.com/w3c/webai/issues/23)

## What It Does

Paste any code into the browser-based checker and get an instant safety analysis. The tool runs entirely client-side -- your code never leaves your browser. No server, no data collection.

It checks for:

- **Dangerous function calls** -- eval(), exec(), os.system(), pickle, subprocess with shell=True, document.write(), innerHTML
- **Hardcoded secrets** -- API keys, passwords, tokens embedded in source code
- **Network access and data exfiltration** -- outbound requests, disabled SSL verification
- **File system risks** -- insecure temp files, permission changes, directory traversal
- **Injection vulnerabilities** -- SQL injection, shell injection, XSS vectors
- **Debug leaks** -- sensitive values in print/console.log statements

Each finding includes the severity level, the affected line, an explanation of the risk, and a concrete fix.

## Supported Languages

The checker supports the three languages where AI-generated code vulnerabilities are most common:

| Language | Rule Count | Key Checks |
|----------|-----------|------------|
| Python | 16 rules | eval/exec, pickle, subprocess shell injection, YAML unsafe load, SQL injection, hardcoded secrets, insecure temp files |
| JavaScript / TypeScript | 14 rules | eval, innerHTML, document.write, new Function, postMessage without origin check, CORS wildcard, child_process, localStorage secrets |
| Shell (Bash) | 11 rules | eval, curl pipe to sh, chmod 777, rm -rf with unquoted vars, missing set -euo pipefail, predictable temp paths, hardcoded credentials |

Language is auto-detected, or you can select manually using the tabs.

## How to Use

1. Go to [https://starborn.github.io/codesafetycheckr2026/](https://starborn.github.io/codesafetycheckr2026/)
2. Paste your code into the editor
3. Click **Analyze Safety**
4. Review the findings -- each one shows severity, line number, explanation, and fix
5. Click **Export Report** to download a text report

### Pre-loaded Examples

The app includes three built-in examples (one per language) demonstrating common vulnerabilities. Click **Load Example** on any language tab to try them:

- **Python example** -- demonstrates eval(), pickle deserialization, os.system() command injection, subprocess with shell=True, hardcoded API key, disabled SSL verification, and SQL injection via f-strings
- **JavaScript example** -- demonstrates eval(), innerHTML assignment, document.write(), new Function() constructor, hardcoded secrets in client code, sensitive data in localStorage, and postMessage listener without origin validation
- **Shell example** -- demonstrates curl piped to sudo bash, chmod 777, rm -rf with unquoted variables, eval of user input, hardcoded passwords, and predictable temp file paths

## Contribute Test Samples

We invite the community to test the checker with your own code -- whether AI-generated or human-written -- and contribute samples and reports back to this repository via Pull Request.

This builds a shared corpus of real-world vulnerability patterns, directly supporting the W3C recommendation ([webai#23](https://github.com/w3c/webai/issues/23)) to develop tools and practices that help AI code assistants generate safer code.

### How to Contribute

1. **Fork this repository**
2. **Test your code** at [the live app](https://starborn.github.io/codesafetycheckr2026/)
3. **Save the code sample** in `samples/` using this naming convention:
   - `samples/python/description-of-sample.py`
   - `samples/javascript/description-of-sample.js`
   - `samples/shell/description-of-sample.sh`
4. **Save the exported report** in `reports/` with a matching name:
   - `reports/description-of-sample-report.txt`
5. **Open a Pull Request** with:
   - A brief description of where the code came from (AI-generated, open source, tutorial, etc.)
   - The safety score the checker assigned
   - Any false positives or missing rules you noticed

### What Makes a Good Contribution

- Code generated by an LLM (Claude, ChatGPT, Copilot, Gemini, etc.) -- especially interesting if it came from a real task
- Code from tutorials or documentation that contains hidden vulnerabilities
- Examples where the checker misses something (false negatives) -- helps us improve the rules
- Examples where the checker flags something incorrectly (false positives) -- also helps
- "Before and after" pairs showing vulnerable code and its fixed version

### Contribution Guidelines

- **Do not include real secrets, credentials, or API keys** -- replace them with obvious placeholders like `sk-EXAMPLE-KEY-12345`
- **Do not include proprietary or confidential code** -- only submit code you have the right to share
- **One sample per file** -- keep samples focused on a specific pattern or vulnerability type
- **Add a comment header** to each sample file:

```python
# Source: [AI-generated / open-source / tutorial / etc.]
# Generator: [Claude / ChatGPT / Copilot / human / etc.]
# Date: YYYY-MM-DD
# Description: Brief description of what this code does and why it was tested
# Safety Score: XX/100
```

## Companion Tool

The **W3C Code Quality Checker** ([Colab notebook](https://github.com/w3c/webai/issues/23)) extends this approach to web standards compliance, checking HTML/CSS/JS against four W3C principles:

- Accessibility (WCAG 2.2)
- Internationalization (W3C I18N)
- Privacy (W3C Privacy Principles)
- Security (OWASP Top 10)

Together, the two tools address both general code safety and web-specific standards compliance.

## Technical Details

- **Static analysis only** -- no code is executed, ever
- **Client-side only** -- runs entirely in the browser, no server component
- **Zero dependencies at runtime** -- single HTML file with embedded React
- **40+ rules** across three languages
- **Scoring** -- 0-100 safety score with weighted severity (critical=25, high=12, medium=5, low=1)

## Context

This tool was developed in response to [w3c/webai#23](https://github.com/w3c/webai/issues/23), which recommends that W3C encourage AI code assistants to generate high-quality code respecting accessibility, internationalization, privacy, and security principles, and to develop tools that help users verify and improve AI-generated code.

Research shows that AI-generated code frequently introduces security vulnerabilities and quality issues that static analysis can detect. This tool provides an accessible, zero-setup way for anyone to check code before deploying it.
